# Research Engine: Execute

Internal execution skill for Perplexity API research. Sub-agents receive a
query, depth, and metadata from :orchestrate, then execute the API call, save
results, and return findings.

## Sub-Agent Tool Constraints

This skill runs inside background sub-agents using the `research-executor`
agent definition (`integrations/research-engine/agents/research-executor.md`), which declares
explicit tools: Read, Write, Bash, Glob, Grep, WebFetch, WebSearch.

**Write permissions** come from the settings.json allow list, NOT from the
agent definition's `permissionMode` (which does not propagate to background
agents). The rule `Edit(/research/**)` in `~/.claude/settings.json`
auto-approves Write operations to `<project-root>/research/`.

**Bash permissions** are also via settings.json. The rule
`Bash(*pplx-curl.sh*)` auto-approves only commands containing "pplx-curl.sh".

See `references/permissions.md` for the full permission architecture,
how to add new rules, and known limitations.

**Rules:**
- The ONLY Bash call in this workflow is `integrations/research-engine/scripts/pplx-curl.sh`
- All file reads: use the **Read** tool
- All file writes: use the **Write** tool
- All file searches: use **Glob** and **Grep** tools
- All web fetching: use **WebFetch** tool
- Directory creation and timestamp generation are handled INSIDE pplx-curl.sh
- Do NOT run mkdir, date, variable assignments, or any other Bash commands

## Input Parameters

The orchestrator dispatches a sub-agent with these parameters:

| Parameter | Required | Description |
|-----------|----------|-------------|
| query | yes | The specific research query string |
| depth | yes | `"quick"` or `"deep"` |
| research_id | yes | Pre-generated by orchestrator (e.g., `RE-2026-0302-001`) |
| source_type | yes | `voice-memo` \| `email` \| `note` \| `manual` |
| source_ref | yes | Filename, email ID, or topic string |
| triggered_by | yes | `daily-brief` \| `on-demand` |
| cascade_chain | if cascade child | Array of `{id, topic, key_finding}` for all ancestors |
| context_from_parent | if cascade child | KEY FINDINGS TO BUILD ON + WHAT TO INVESTIGATE |

## Output Directory Structure

All output goes under the current project root: `./research/`

Research travels with the project. The orchestrator's working directory is
inherited by sub-agents, so files land in the correct project automatically.

Directory creation is handled by `pplx-curl.sh --research` mode.

```
research/
├── raw/                              # Complete API responses (never truncated)
│   └── 20260302_143022_topic-slug.json
├── sources/                          # Full article content (deep dives only)
│   └── 20260302_sparktoro-com_prompt-diversity.md
├── cascades/                         # Follow-up requests for orchestrator
│   └── RE-2026-0302-001-cascade.yml
└── topic-slug-2026-03-02.md          # Thread file with YAML frontmatter
```

## API Key

Handled internally by `integrations/research-engine/scripts/pplx-curl.sh`. Do NOT extract the key
manually, use the Read tool on the key file, or run any Bash command to read it.

## Model Selection

Binary decision:

- **Exhaustive, report-style research?** -> `sonar-deep-research`
- **Everything else** -> `sonar-reasoning-pro` (default)

For RCA/debugging queries: sonar-reasoning-pro is mandatory (causal reasoning
required). See `references/rca-workflow.md`.

For multi-model synthesis patterns: see `references/models.md`.

## Auto-Select Parameters

**search_mode** (based on query content):
- Studies, papers, peer-reviewed, scientific -> `"academic"`
- Companies, filings, earnings, SEC, financial -> `"sec"`
- Everything else -> `"web"`

**return_images** (based on query content):
- Product research, visual topics, places, design -> `true`
- Technical, analytical, factual -> `false`

**date_filters** (auto-suggest based on query content):
- Query contains temporal signals ("recent", "latest", "2025", "2026",
  "last month", "since January", "this year") -> add
  `"search_after_date_filter"` set to 6 months before today's date
  (format: `"MM/DD/YYYY"`)
- No temporal signal -> omit

**domain_filters** (auto-suggest based on query content):
- Query targets a specific ecosystem -> add `"search_domain_filter"` array:
  - Python: `["docs.python.org", "github.com", "pypi.org"]`
  - JavaScript/Node: `["developer.mozilla.org", "github.com", "npmjs.com"]`
  - Apple/iOS: `["developer.apple.com", "github.com"]`
  - Legal/regulatory: `[".gov"]`
  - Academic: `[".edu", "scholar.google.com"]`
- General query -> omit

For all available API parameters: see `references/api-reference.md`.

## Quick Scan Workflow

### Step 1: Plan the Call

Using LLM reasoning only (NO Bash, NO tool calls), determine:

1. **topic_slug**: Lowercase, hyphens, no spaces, derived from query
   (e.g., "app-store-accountability-act")
2. **model**: Per the binary decision above (sonar-reasoning-pro or
   sonar-deep-research)
3. **system_prompt**: Based on query context:
   - **Software/technical**: "Prioritize official documentation, GitHub repositories, and high-quality technical sources. Include code examples when relevant. Note version compatibility."
   - **Academic/scientific**: "Prioritize peer-reviewed sources and reputable publications. Note methodology limitations. Highlight conflicting findings."
   - **Fact-checking**: "Cross-reference claims against multiple independent sources. Distinguish verified facts from disputed claims. Rate confidence levels."
   - **Business/market**: "Prioritize authoritative sources (SEC filings, official reports). Include quantitative data. Note potential biases."
   - **RCA/debugging**: Use the RCA system prompt from `references/rca-workflow.md`.
4. **search_mode**: Per auto-select rules above
5. **return_images**: Per auto-select rules above
6. **user_message**: The query string. If this is a **cascade child**, prepend
   the `context_from_parent` so the model builds on prior findings.

Construct the full JSON payload as a string. Escape internal double quotes
with backslash for Bash argument passing.

### Step 2: Execute API Call (SINGLE Bash Call)

This is the ONLY Bash command in the entire workflow. No other Bash calls
are permitted.

```bash
integrations/research-engine/scripts/pplx-curl.sh --research "TOPIC_SLUG" '{"model":"MODEL_NAME","messages":[{"role":"system","content":"SYSTEM_PROMPT"},{"role":"user","content":"USER_QUERY"}],"web_search_options":{"search_context_size":"high"},"search_mode":"SEARCH_MODE","return_related_questions":true,"return_images":RETURN_IMAGES,"temperature":0.2,"stream":false}'
```

The `--research` mode handles internally:
- Creating `research/raw/`, `sources/`, `cascades/` directories
- Generating a timestamp for the filename
- Saving the API response to `research/raw/TIMESTAMP_TOPIC-SLUG.json`
- Printing the output file path as `OUTPUT_PATH=<path>` (last line of output)

Parse the last line of output to get the file path for Step 3.

**For deep research, add to the JSON body:**
```json
"reasoning_effort": "high"
```

**CRITICAL**: `search_context_size` MUST be nested inside `web_search_options`.
Top-level placement is silently ignored (falls back to "low"). This is the
single most common API mistake.

### Step 3: Read and Extract

Use the **Read** tool to retrieve the saved JSON file. For large responses
(>2000 lines), use offset/limit to read in chunks.

Extract from the response:
- Content: `.choices[0].message.content`
- Sources: `.search_results[]` (title, URL, date, snippet) -- use this, not `citations`
- Cost: `.usage.cost.total_cost`
- Token count: `.usage.total_tokens`
- Related questions: `.related_questions` (if present)

### Step 4: Write Thread File

Use the **Write** tool to create `research/[topic-slug]-YYYY-MM-DD.md`
with YAML frontmatter.

**YAML Frontmatter**:

```yaml
---
research_id: RE-2026-0302-001
source_type: voice-memo
source_ref: "Multi-Agent Research Framework.md"
depth: quick
triggered_by: daily-brief
date: 2026-03-02
queries: 1
models_used: [sonar-reasoning-pro]
cost_estimate: $0.02
cascade_parent: null
---
```

For cascade children, set `cascade_parent` to the parent's research_id.

**Thread body format**:

```markdown
# [Topic Name]

## Query 1: [Brief query description]
**Timestamp**: YYYY-MM-DDTHH:MM:SSZ
**Raw**: [raw/TIMESTAMP_topic-slug.json](raw/TIMESTAMP_topic-slug.json)
**Model**: sonar-reasoning-pro | **Tokens**: N | **Cost**: $X.XX

### Findings
[Synthesized findings -- key points, not full dump]

### Sources
[1] Title - domain.com
    URL | Published: YYYY-MM-DD
[2] Title - domain.com
    URL | Published: YYYY-MM-DD

### Related Questions
- [From API response, if present]

## Synthesis (Updated: YYYY-MM-DD)

### Key Conclusions
- [Conclusion 1]
- [Conclusion 2]

### Open Questions
- [What remains unresolved]

### Confidence Assessment
- High confidence: [topics]
- Moderate confidence: [topics]
- Needs verification: [topics]
```

### Step 5: Return Summary to Orchestrator

Return a structured summary that the orchestrator can use:
- research_id
- topic (slug)
- key_findings (3-5 bullet points)
- source_count
- cost
- thread_file_path
- raw_file_path
- confidence (high/medium/low)
- gaps (anything the research could not answer)

## Deep Dive Workflow

Follow quick scan Steps 1-4. Do NOT execute quick scan Step 5 (Return Summary).
Instead, continue with these steps:

### Step 5: Read Primary Sources

For each cited source URL from the Perplexity response, fetch and read the
full content. Choose the fetch method based on URL type:

**For PDF URLs** (URL ends in `.pdf` or Perplexity indicates it's a document):
1. Download using `pplx-curl.sh --fetch-pdf "<URL>"` (single Bash call,
   saves to `research/sources/`)
2. Parse the output: `LOCAL_PATH=<path>` on success, non-zero exit on failure
3. On success: use the **Read** tool with the local path to extract content
   (supports PDF parsing with page ranges for large documents)
4. On failure (exit 1 = HTTP error, exit 2 = not a PDF): record the URL and
   failure reason in a `blocked_sources` list for the cascade file (Step 7)

**For YouTube URLs** (youtube.com/watch, youtu.be):
1. Extract the transcript using yt-dlp (must be installed: `pip install yt-dlp` or `brew install yt-dlp`). If yt-dlp is not installed, guide the user through installation.
   ```bash
   yt-dlp --write-auto-sub --sub-lang en --skip-download --print-to-file "%(title)s" /dev/stderr -o "%(id)s" "VIDEO_URL" 2>&1
   ```
   Or for structured JSON output:
   ```bash
   yt-dlp --dump-json --skip-download "VIDEO_URL"
   ```
2. Extract title, channel, transcript text, and view count from the output
3. Use the **Write** tool to save to
   `research/sources/[timestamp]_youtube_[video-id].md`
   with a header noting the video title, channel, and view count
4. Treat the transcript as a primary source (same as fetched articles)

**For HTML URLs** (everything else):
1. Fetch the article using **WebFetch** with a prompt asking for the full
   content relevant to the research question
2. Use the **Write** tool to save to
   `research/sources/[timestamp]_[domain]_[slug].md`

For each source (PDF or HTML), note findings:
   - Relevant evidence for the research question
   - Contradictions to Perplexity's findings
   - Notable findings outside research scope
   - Source quality assessment
   - **Leads** -- URLs or references cited in the article worth following

**Wave pattern**: After processing all sources, collect leads. If meaningful
leads exist, fetch them with WebFetch (or --fetch-pdf for PDFs). Repeat until
the question is answered or no new leads remain.

### Step 6: Synthesize

Integrate all source findings with Perplexity's response:
- Evidence Perplexity missed or understated
- Contradictions resolved
- Confidence assessed by source quality
- Leads followed across waves

Update the thread file with a "Primary Source Findings" section and "Leads
Followed" section.

### Step 7: Assess Completeness

If critical gaps remain after primary source reading, use the **Write** tool
to create a cascade request file at `research/cascades/`.

**Cascade Request Format** (`research/cascades/[research_id]-cascade.yml`):

```yaml
parent_research_id: RE-2026-0302-001
parent_topic: "multi-agent research systems"
parent_summary: |
  Research found three major frameworks but could not determine
  which handles cascading delegation best.

cascade_chain:
  - id: RE-2026-0302-001
    topic: "multi-agent research systems"
    key_finding: "Three frameworks identified, delegation unclear"

follow_ups:
  - topic: "LangGraph vs CrewAI cascading delegation"
    why: |
      Parent research identified both as leading frameworks but
      sources disagreed on multi-level delegation support.
    suggested_query: "LangGraph vs CrewAI autonomous sub-agent delegation cascading comparison 2026"
    context_for_next_agent: |
      KEY FINDINGS TO BUILD ON:
      - LangGraph uses graph-based state machine (source: LangChain docs)
      - CrewAI uses role-based agent definitions (source: CrewAI GitHub)
      WHAT TO INVESTIGATE:
      - Can either handle agent A spawning agent B spawning agent C?
      - What are the context-passing mechanisms?
    recommended_depth: deep
```

If any sources could not be fetched (PDF download failed, site blocked curl),
add a `blocked_sources` section to the cascade file:

```yaml
blocked_sources:
  - url: "https://www.kslegislature.gov/li/b2025_26/measures/documents/sb372_00_0000.pdf"
    reason: "exit 2: downloaded file is not a PDF (got text/html)"
    what_it_contains: "Kansas SB 372 full bill text with enforcement provisions"
```

The orchestrator uses this to attempt fetching with chawan or playwright-cli
(which have full browser capabilities) before dispatching cascade children.

Only write a cascade request when:
- A critical sub-question could not be answered
- Sources explicitly disagreed and resolution requires focused research
- A lead from primary sources opens a significant new angle
- Sources were blocked and their content is needed to answer the question

Do NOT cascade for minor gaps or tangential curiosity.

### Step 8: Return Summary to Orchestrator

Same as quick scan Step 5, plus:
- primary_sources_read (count)
- waves_completed (count)
- cascade_requested (boolean)
- cascade_file_path (if applicable)

Update the YAML frontmatter to reflect final state (queries count, models used,
total cost).

## Error Handling

Since :execute runs as a sub-agent (no user interaction), handle errors
decisively. The goal is to maximize Perplexity usage and minimize silent
fallbacks.

### HTTP Errors (non-zero exit from pplx-curl.sh)

Check stderr for the error:

- HTTP 429 (rate limited): Wait 10 seconds, retry the same pplx-curl.sh call once
- HTTP 500 (server error): Retry once after 5 seconds
- HTTP 401 (unauthorized): Abort -- API key issue requires human intervention
- HTTP 400 (bad request): Attempt to reformulate (remove special characters,
  shorten query), retry once
- All other failures after retry: Abort and return error details to orchestrator

### Irrelevant Results (Perplexity call succeeded but content is off-topic)

If the API call returns successfully but the content does not address the query
(e.g., date filters excluded all relevant results, domain filters were too
narrow, or the model misunderstood the query):

1. **Diagnose the cause** by checking which parameters may have constrained
   results too aggressively:
   - Date filters excluding relevant older content → remove or widen date range
   - Domain filters too narrow → remove domain filters
   - Query too specific → broaden the query while keeping the core intent
   - search_mode mismatch (e.g., "academic" for a practitioner topic) → switch
     to "web"
2. **Retry with adjusted parameters** — this is a NEW pplx-curl.sh call (the
   second Bash exception beyond ID generation). Save to a separate raw file.
3. If the retry also returns irrelevant results, THEN fall back to
   WebSearch/WebFetch (see below).

### Fallback to WebSearch/WebFetch

Use ONLY when:
- pplx-curl.sh is completely inaccessible (Bash denied), OR
- Perplexity returned irrelevant results after one retry with adjusted parameters

Fallback produces lower-quality results (no model selection, no cost tracking,
no related questions, no reasoning trace).

**When falling back, the sub-agent MUST:**

1. Set `models_used` to include `web-search-fallback` in the YAML frontmatter
2. Add a `fallback_reason` field to the YAML frontmatter explaining why
   Perplexity failed (e.g., `"date filter excluded all results; retry with
   wider range also returned irrelevant content"`)
3. Add `⚠️ FALLBACK` prefix to the thread file title (e.g.,
   `# ⚠️ FALLBACK: Topic Name`)
4. Include in the return summary to orchestrator:
   - `fallback_used: true`
   - `fallback_reason: "<reason>"`
   - `perplexity_attempts: N` (how many Perplexity calls were tried)
   - `retry_params_used: "<what was changed>"` (e.g., "removed date filter")
